# PROMPTS.md

## AI Summary Generation Prompt

```txt
You are an AI financial optimization assistant.

Analyze the user's AI tooling stack and generate a concise personalized audit summary in around 100 words.

Context:
- Team Size: {teamSize}
- Primary Use Case: {primaryUseCase}
- Current Monthly Spend: ₹{totalMonthlySpend}
- Estimated Monthly Savings: ₹{totalMonthlySavings}
- Estimated Annual Savings: ₹{totalAnnualSavings}

Tool Breakdown:
{toolBreakdown}

Instructions:
- Be honest and realistic.
- Do not exaggerate savings.
- If savings are low, mention the stack already looks optimized.
- If savings are high, mention optimization opportunities clearly.
- Mention duplicate tooling or over-provisioning if applicable.
- Keep the tone professional and helpful.
- Keep response under 100 words.
- Do not use bullet points.
```

---

# Why I Wrote the Prompt This Way

I wanted the AI summary to feel realistic instead of sounding like generic marketing text. The assignment specifically mentioned that the audit logic itself should remain rule-based, so I only used AI for summarization and explanation.

I added clear instructions like:
- “Do not exaggerate savings”
- “Be honest and realistic”
- “Keep under 100 words”

because large language models sometimes overstate optimization opportunities even when the savings are small.

I also included structured context like:
- team size
- use case
- savings
- per-tool recommendations

so the summary feels personalized instead of random.

The tool breakdown section was important because it allowed the AI to reference actual recommendations already generated by the audit engine.

---

# What I Tried That Didn’t Work

## 1. Very Short Prompts

Initially I tried using a simple prompt like:

```txt
Summarize this AI spend audit.
```

The outputs were too generic and repetitive. Most summaries sounded almost identical regardless of the audit result.

---

## 2. Allowing Fully Creative Responses

At one point I allowed the model to freely explain optimization opportunities without restrictions.

This caused problems:
- exaggerated savings language
- fake assumptions about company scale
- recommendations not supported by the audit engine

Because of this, I switched to a more constrained prompt structure.

---

## 3. Large Structured JSON Prompts

I also experimented with sending the full audit object as raw JSON.

This increased token usage unnecessarily and sometimes made the summaries less readable because the model focused too much on raw data formatting instead of writing a clean explanation.

I replaced this with smaller structured text sections instead.

---

# Fallback Strategy

If the Groq API fails or rate limits the request, the application falls back to a templated summary generated directly in the frontend.

This ensures:
- audit generation never breaks
- users always receive a usable result
- the UI remains responsive even during API failures
